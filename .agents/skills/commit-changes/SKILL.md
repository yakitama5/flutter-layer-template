# Commit Changes

## ワークフロー

1. リポジトリの `AGENTS.md` とコミットに関するローカル規約を確認する。
2. `git status --short`、`git diff`、`git diff --cached` で変更全体を把握する。
3. ユーザーが意図した変更だけを選び、独立した目的が混在する場合はコミットを分ける。
4. リポジトリが要求する生成、静的解析、テストを実行する。失敗した場合は原則としてコミットせず、結果を報告する。
5. 対象パスを明示してステージする。ユーザーの別作業や未追跡ファイルを含めない。
6. `git diff --cached --check` と `git diff --cached` でステージ済み差分を最終確認する。
7. 変更の主目的に対応する種類を一つ選び、規約どおりのメッセージでコミットする。
8. `git status --short` と直近のコミットを確認し、コミット ID、メッセージ、検証結果、残った変更を報告する。

`git add -A` や `git add .` は避け、対象パスを列挙する。既存変更の破棄、履歴の書き換え、フックの無効化、秘密情報のコミットは行わない。コミットフックでファイルが変化した場合は、差分を再確認してから扱う。

## コミットメッセージ

件名を `<emoji><type>: <summary>` の形式にする。`emoji` と `type` の間に空白を入れず、`type` は次の表記をそのまま使う。`summary` は変更内容が判別できる簡潔な日本語とし、末尾に句点を付けない。

| 種類 | 用途 |
|---|---|
| `🎉new` | プロジェクトや大きな土台の新規作成 |
| `🔖tag` | バージョンやタグ、リリース識別子の更新 |
| `✨feat` | 利用者に見える機能や新しい能力の追加 |
| `🐛fix` | 不具合の修正 |
| `♻️refactor` | 振る舞いを変えない内部構造の改善 |
| `📚doc` | ドキュメントのみの変更 |
| `🎨design` | UI、見た目、スタイルの変更 |
| `🐎perf` | 性能の改善 |
| `🔧chore` | 保守、設定、依存関係、開発環境の変更 |
| `🚨test` | テストのみの追加や修正 |
| `🗨️comment` | コメントのみの追加や修正 |
| `♻️generete` | 自動生成物のみの生成や更新 |
| `🚧wip` | 未完了作業の一時保存。ユーザーが明示した場合だけ使う |
| `💩depre` | 非推奨化、旧機能の廃止に向けた変更 |

複数の種類に該当するときは、変更の主目的で選ぶ。たとえば、機能追加に伴うテストやドキュメントは `✨feat` にまとめ、テストだけの改善は `🚨test` にする。自動生成物がソース変更に付随する場合はソース側の種類を使い、生成物だけの更新に限って `♻️generete` を使う。

例:

```text
✨feat: コミット用スキルを追加
🐛fix: ログイン失敗時の画面遷移を修正
♻️generete: モデルの生成コードを更新
```
